# Pebble MCP Server ‚Äî Design Spec

A cozy virtual pet MCP server running on Cloudflare Workers. Inspired by Travel Frog ‚Äî an idle game where your pet lives its own life, goes on trips, and sends back stories. You prepare its pack and check in through a feed.

## Architecture

### Approach: Lazy Evaluation (Compute on Read)

No background processing. When the player checks in via the `feed` tool, the **Life Engine** computes what the pet has been doing since the last check. This is deterministic ‚Äî the same check at the same time always produces the same history.

### Worker: `workers/pebble`

A new worker in the monorepo following the same patterns as `workers/auth`:

- **Hono** for HTTP routing
- **@modelcontextprotocol/sdk + agents** for MCP server
- **D1** (shared `rs-db` database) via **Drizzle ORM**
- **OAuth** via existing `rs-auth` worker
- **No server-side AI** ‚Äî the player's MCP client (Claude Code, Cursor, Codex) generates narratives and ASCII art. The server returns structured data only.

Future chat API endpoints will be added to the same worker (deferred ‚Äî not in scope for initial build).

## MCP Tools

| Tool | Description |
|---|---|
| `adopt` | Create your pet. Choose a name. Rolls random personality traits. Returns structured data ‚Äî the client LLM generates the ASCII art and welcome narrative. Stores the client-generated ASCII art via `save-story`. One pet per player. |
| `feed` | Main interaction ‚Äî "open the app." Triggers Life Engine computation, returns structured event data (activity types, locations, items found, encounters). The client LLM narrates the events. Also collects pending item rewards and shows current pet status. |
| `pack` | Set out items from inventory for the pet's next trip. The pet chooses from the pack based on its personality when it departs. |
| `items` | View inventory ‚Äî owned items. |
| `save-story` | Called by the client LLM after narrating events. Saves the generated narrative text back to the `story` table for persistent history. Also saves pet ASCII art on first adoption. |

## Data Model

### `pet` table

| Column | Type | Description |
|---|---|---|
| `id` | text PK | Unique pet ID |
| `player_id` | text FK ‚Üí player | Owner |
| `name` | text | Player-chosen name |
| `seed` | integer | Random seed for deterministic life simulation |
| `ascii_art` | text nullable | Unique ASCII art (generated by client LLM at adoption, saved via `save-story`) |
| `curiosity` | integer (0-100) | Explores new places vs. stays familiar |
| `energy` | integer (0-100) | Active (adventuring) vs. passive (sleeping) |
| `sociability` | integer (0-100) | Likelihood of visiting places where other pets go |
| `courage` | integer (0-100) | Visits rare/far locations vs. stays close |
| `creativity` | integer (0-100) | Tendency to craft or create souvenirs |
| `last_checked_at` | integer | Timestamp of last feed check (compute-from marker) |
| `created_at` | integer | Birth timestamp |

Personality traits are randomly rolled at adoption and immutable. They directly weight the Life Engine's activity probability distribution.

### `item` table (catalog)

| Column | Type | Description |
|---|---|---|
| `id` | text PK | Item ID |
| `name` | text | Display name |
| `description` | text | Flavor text |
| `ascii_art` | text | Item ASCII art |
| `category` | text | `food`, `tool`, `charm`, `souvenir` |
| `effect_target` | text nullable | Which personality trait it modifies |
| `effect_strength` | integer nullable | How much it shifts the trait |
| `rarity` | text | `common`, `uncommon`, `rare` |

### `inventory` table

| Column | Type | Description |
|---|---|---|
| `id` | text PK | Row ID |
| `pet_id` | text FK ‚Üí pets | Owner pet |
| `item_id` | text FK ‚Üí items | Item |
| `quantity` | integer | How many owned |

### `pack` table

| Column | Type | Description |
|---|---|---|
| `pet_id` | text FK ‚Üí pets | Owner pet |
| `item_id` | text FK ‚Üí items | Item set out |
| `quantity` | integer | How many packed |

When the pet departs, it selects from the pack based on personality. Consumed items are removed.

### `story` table

| Column | Type | Description |
|---|---|---|
| `id` | text PK | Row ID |
| `pet_id` | text FK ‚Üí pets | The pet |
| `time_window` | integer | Which 30-min slot (timestamp) |
| `activity_type` | text | `sleeping`, `eating`, `wandering`, `traveling`, `crafting`, etc. |
| `location` | text nullable | Where the pet is (null = home) |
| `story` | text nullable | Client LLM narrative (saved via `save-story` after generation) |
| `encountered_pet_id` | text nullable FK ‚Üí pets | Another pet met at this location |
| `items_found` | text nullable | JSON array of item IDs found/crafted |
| `collected` | integer | 0 = rewards pending, 1 = collected by player |
| `created_at` | integer | When this log was computed |

This is a write-behind cache. Entries are computed on `feed` and stored so they don't need recomputation.

## Life Engine

The core simulation algorithm, triggered by `feed`:

```
1. Read pet.last_checked_at ‚Üí compute elapsed time to now
2. Divide elapsed time into 20-minute time windows
3. For each window:
   a. Seed RNG: hash(pet.seed, window_timestamp)
   b. Read pack items ‚Üí compute temporary trait modifiers
   c. Weight activity probabilities by modified traits
   d. Determine pet state:
      - If pet was home: chance to depart on trip (energy + courage)
      - If pet was traveling: chance to arrive at location, continue, or return home
      - If pet is home: pick home activity (sleeping, eating, wandering, crafting)
   e. If at a location: check if another pet's RNG puts them at the same
      location in the same window ‚Üí passive encounter
   f. If traveling: determine items found based on location + traits
   g. If creative enough: chance to craft an item
4. Store computed activities in story table (structured data, no narrative yet)
5. Update pet.last_checked_at to now
6. Return: current state, structured event data, previously saved narratives, pending rewards
7. Client LLM narrates new events, then calls `save-story` to persist the narratives
```

### Deterministic RNG

All randomness is seeded by `hash(pet.seed, time_window)`. This means:

- The same pet always does the same thing in the same time window
- No background processing needed
- State can be recomputed if the cache is lost

### Encounter Detection

When computing a pet's activities, check other pets' computed locations for the same time window. If locations match, it's a passive encounter. Since RNG is deterministic, both owners will see the same encounter when they each check in.

For performance: only check pets that have overlapping travel windows. Index `story` on `(time_window, location)`.

### Trip Lifecycle

1. Pet is home ‚Üí Life Engine rolls "depart" based on energy + courage + whether pack has items
2. Pet departs ‚Üí selects items from pack (personality-driven), pack items consumed
3. Pet travels for N windows (duration based on energy + items taken)
4. During trip: visits locations, may encounter other pets, finds items
5. Pet returns home ‚Üí souvenirs + stories added to inventory/feed

## Story Feed

Every event in the pet's life becomes a story entry in the feed: departures, arrivals, encounters, crafting, returns home. The feed is a chronological log of everything that happened.

When the player calls `feed`:

1. Life Engine computes all events since last check ‚Üí stored as structured data in `story` table
2. Returns structured event data to the client LLM (activity type, location, items, encounters)
3. For older events, returns the previously saved narrative from the `story` column
4. Events that yielded items show as **collectible** ‚Äî calling `feed` collects pending rewards
5. The client LLM narrates the new events in a cozy tone, then calls `save-story` to persist

**Client-side generation flow:**
- `feed` returns: `{ events: [{ id, type, location, items, hasNarrative: false }, ...] }`
- Client LLM sees the structured data and generates warm, cozy narratives
- Client calls `save-story` with `{ stories: [{ id: "story-id", narrative: "Your pet curled up..." }, ...] }`
- Next `feed` call returns the saved narratives for those events

Collectible items are tracked via `story.collected` (boolean, default false).

## ASCII Art

- **Pet**: unique abstract creature generated by the client LLM at adoption. Saved to `pet.ascii_art` via `save-story`. Each client may produce different art ‚Äî that's part of the charm.
- **Items**: each item has small ASCII art defined in the item catalog (server-side static data).

## Authentication

Reuse the existing OAuth flow from `workers/auth`. The pet worker registers as an MCP server with OAuth protection, same pattern as `rs-auth-mcp`. User identity comes from the OAuth token's `userId` prop.

## Scope Boundaries

### In scope (initial build)

- 5 MCP tools: adopt, feed, pack, items, save-story
- Life Engine with lazy evaluation
- Deterministic pet behavior driven by personality traits
- Passive encounters between pets
- Structured event data ‚Üí client LLM narrates ‚Üí save-story persists
- No server-side AI ‚Äî leverages the player's MCP client for generation
- Item system: pack, souvenirs
- OAuth authentication via existing rs-auth

### Deferred

- Chat API (REST endpoints for non-MCP clients)
- Home decoration system
- Visit other users' pets
- Friendship/social features beyond passive encounters
- Real-time notifications
